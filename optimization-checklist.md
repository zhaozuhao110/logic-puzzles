# 待优化项清单（细化版）

## 1. 使用方式

- `P0`：必须优先完成，直接影响页面结构、可访问性、SEO 和后续维护成本。
- `P1`：建议第二批完成，主要提升转化、阅读效率和站内串联。
- `P2`：可在主干问题收敛后补充，主要增强体验与内容表现力。
- 验收原则：每个页面完成后，至少检查语义结构、移动端首屏、键盘操作、推荐链路、标题与副标题是否可读。

## 2. 统一先行项

### 全站公共整改

- `assets/base.css` : P0 统一整理颜色变量、间距变量、按钮态、卡片态和标题层级；消除页面各自定义同类样式的情况；补一套移动端断点规范。验收：常见按钮、卡片、标签、结论块不再在页面内重复定义。
- `assets/problem-gold.css` : P0 抽象题解页公共骨架样式，至少覆盖题干区、步骤区、结论区、延伸区、面包屑、上一篇下一篇、相关推荐；P1 增加表格页和交互页的扩展样式钩子。验收：通用题页改造时只需要补少量页面级样式。
- `assets/problem-stepper.js` : P0 为步骤卡补充键盘可达性、当前步骤状态、禁用态和读屏语义；P1 支持深链接到指定步骤和返回上次阅读位置。验收：Tab 可聚焦步骤入口，方向操作或回车可切换，当前步骤对读屏器可感知。
- 全站元信息 : P0 为首页、目录页、题页统一标题模板、`meta description` 生成规则和社交分享摘要；P1 增加结构化数据预留。验收：所有页面都不再使用 slug 作为对用户可见的副标题。

### 题目主题化表达规范

- 设计基准 : 以 `problems/soda-bottle.html` 作为“题意深度契合型页面”参考，不是简单换色，而是把题目机制翻译成页面视觉、动效和交互语言。
- P0 每题必须具备专属主题语义 : 每个题目都要提炼 1 个核心意象和 1 组主题色，例如帽子题突出遮挡、视角、阵列，天平题突出左右平衡、重量差，海盗题突出宝箱、投票、航海感。验收：用户不看正文，只看首屏也能感受到题目的世界观或机制。
- P0 首屏必须服务题意 : Hero 区不再只放标题和副标题，要像 `soda-bottle.html` 一样用图形、关键词、题目摘要卡或主图，让题目机制先被“看懂”。验收：首屏元素不是装饰，而是帮助用户理解题目。
- P1 内容模块必须与题目机制联动 : 数学题可用过程模拟、数量递减、公式卡；空间题可用结构示意、视角切换；博弈题可用回合卡、角色卡、策略分支；真假话题可用线索表、身份状态切换。验收：模块形式和题型匹配，而不是全站一套壳子套到底。
- P1 微动效必须克制且有意义 : 动效优先表现气泡上浮、天平偏移、灯泡亮灭、帽子遮挡、路线推进等“题目内部动作”，避免与题意无关的炫技动画。验收：任何动画都能回答“它在辅助理解什么”。
- P1 建立主题变量层 : 在公共 Token 之外，允许每页定义极少量专属变量，例如 `--theme-primary`、`--theme-glow`、`--theme-icon`、`--theme-pattern`，用于承载题目风格。验收：页面有个性，但不会破坏全站统一骨架。
- P2 建立题型主题库 : 后续可沉淀“数学推导型”“博弈决策型”“空间想象型”“真假话型”“互动模拟型”五类主题化设计模板，减少每次从零设计。验收：后续新题能够在统一规则下快速做出有题感的页面。

## 3. 页面级整改方案

### 首页与总索引

- `index.html` : P0 明确为“品牌首页 + 题库入口”，首屏只保留价值主张、核心卖点和主行动按钮，难度说明与筛选下沉到列表区；P0 补 `main`、`nav`、搜索输入标签、结果数和空状态；P1 将题目卡片按题型、难度、是否互动分组并增加推荐入口；P1 抽离当前内联样式和数据读取逻辑，避免首页成为样式与脚本堆积点；P2 增加“如何开始”与“新手推荐 5 题”。验收：用户进入首页 5 秒内能知道网站价值、如何开始、去哪里做题。
- `outline.html` : P0 明确为“全站总索引页”，不再与首页重复承担品牌介绍；P0 补 `main`、目录导航语义、清晰标题层级与分组锚点；P1 增加搜索、快速筛选、题型聚合、学习路径与专题入口；P1 与首页共用数据源、排序规则和分类规则；P2 增加“按推理类型浏览”“按耗时浏览”视图。验收：目录页承担全量浏览职责，首页承担转化职责，两者边界清晰。

### 模板 A：通用步骤推进题页

- 适用文件 : `problems/age-puzzle-1.html`、`problems/age-puzzle-2.html`、`problems/bacteria-split.html`、`problems/bird-train-distance.html`、`problems/birthday-deduction.html`、`problems/clock-hands-coincide.html`、`problems/eleven-disciples.html`、`problems/fruit-baskets.html`、`problems/guess-number-144.html`、`problems/hat-color-1.html`、`problems/hat-color-2.html`、`problems/hat-color-3.html`、`problems/last-athlete-standing.html`、`problems/lateral-thinking-rent.html`、`problems/monkey-bananas.html`、`problems/pill-weighing.html`、`problems/portia-ii-portrait-1.html`、`problems/portia-ii-portrait-2.html`、`problems/portia-iii-portrait-2.html`、`problems/portia-iii-portrait-3.html`、`problems/six-banknotes.html`、`problems/trapped-island.html`、`problems/turkish-merchant-hats.html`、`problems/will-cows-1.html`、`problems/will-cows-2.html`
- P0 结构统一 : 页面统一改为“题干摘要 -> 分步推理 -> 最终结论 -> 延伸思考”四段；补 `main`、规范 `h1/h2` 层级、`meta description`、题型标签和难度标签。
- P0 文案统一 : 副标题不再直接展示 slug，改成“题型 + 核心解法 + 适合人群”形式，例如“年龄推理 | 方程建模解法 | 适合入门”。
- P0 主题统一 : 在公共骨架上叠加题目专属视觉语义，至少补 1 个题目主意象、1 组专属点缀色、1 处首屏辅助理解图形，不能只保留文字标题。
- P0 交互统一 : 步骤卡改为可聚焦控件，补当前步骤状态、禁用态、读屏文案和点击热区；将“第几步 / 共几步”放在固定位置。
- P1 导航统一 : 页首补面包屑，页尾补上一篇、下一篇、同类推荐和返回题库入口；相关推荐优先同题型、同难度。
- P1 内容统一 : 每页补“本题关键洞见”“常见误区”“一句话总结”，避免页面只剩流程没有复盘。
- P1 主题深化 : 根据题目内容增加一类专属模块，例如年龄题加时间轴或人物关系卡，帽子题加视角阵列，路程题加轨迹示意，猴子香蕉题加移动路径或状态变化图。
- P2 SEO 统一 : 为每页补摘要首段、推荐阅读锚文本、题型关键词的自然语言描述。
- 验收标准 : 首屏能看到题目是什么、难点是什么、如何开始，并能感知题目氛围；步骤区可连续阅读；结尾能继续跳转，不形成阅读死角。

### 模板 B：摘要 + 推理解法型题页

- 适用文件 : `problems/balance-scale.html`、`problems/black-hat-party.html`、`problems/burning-rope.html`、`problems/five-prisoners.html`、`problems/gold-bar.html`、`problems/guess-card.html`、`problems/guigu-apprentices.html`、`problems/king-prophet.html`、`problems/monty-hall.html`、`problems/pirate-gold.html`、`problems/soda-bottle.html`、`problems/three-bulbs.html`、`problems/twelve-balls.html`、`problems/wine-ladle.html`
- P0 首屏重构 : 首屏拆成“题目摘要卡 + 是否展开完整推理”的双层结构，先让用户理解题意，再进入长推理内容。
- P0 基础语义 : 补 `main`、`meta description`、题型标签和结论块，统一标题、副标题、标签、按钮的视觉规则。
- P0 主题表达 : 参考 `problems/soda-bottle.html`，每页增加与题目机制高度契合的主题包装，例如天平题用左右失衡与砝码语义，海盗题用宝箱与投票感，灯泡题用亮灭状态与电路感。
- P1 阅读体验 : 将长段落拆成步骤块、提示块、结论块、误区块；如果有表格或列表，优先改成移动端友好布局。
- P1 串联链路 : 页尾补相关推荐、前后跳转和“同类题继续练习”。
- P1 无障碍 : 所有按钮、折叠区、表格说明、图示说明都要可键盘操作并有状态提示。
- P2 表达增强 : 可补“你也可以先想 30 秒”“适合先做/后做哪些题”的学习引导，并增加题目意象插画、状态图或轻量模拟。
- 验收标准 : 页面不再一上来就是大段正文，用户可以按摘要 -> 推理 -> 结论的顺序自然进入，且视觉表现与题目本身强关联。

### 模板 C：内联样式抽离专项

- 适用文件 : `problems/portia-iii-dagger.html`、`problems/portia-portrait-1.html`、`problems/portia-portrait-2.html`、`problems/ring-rolling-rotation.html`、`problems/three-people-hotel.html`、`problems/tuition-paradox.html`

- P0 清理目标 : 避免同类颜色、边距、卡片、按钮在多个页面中手写复制，降低后续统一换肤和移动端适配成本。
- P1 扩展机制 : 如果页面确实有独特视觉元素，只保留极少量页面级类名，不保留匿名内联样式。
- 验收标准 : 页面 HTML 以结构为主，样式定义不再压在正文中；后续修改间距、配色、标题样式时可以在公共样式文件内完成。

### 轻量题页专项

- `problems/ballerina-mirror-room.html` : P0 补 `main`、`meta description`、题型关键词副标题；P1 增加一张示意图或二维到三维的理解过渡图；P1 补“常见误区”和“换个角度理解”；P2 增加相关推荐，避免页面过薄。验收：页面虽轻，但不空，且首次阅读不会因为缺图而卡住。

### 交互演示页专项

- `problems/blind-socks.html` : P0 将结构拆成题干区、解析区、演示区、结论区，避免演示内容和题目内容并列抢焦点；P0 在演示区前增加“先读题 -> 再看规则 -> 最后操作”的引导文案；P0 修复 SVG `id` 可能重复的问题，改为局部唯一命名或运行时命名空间；P1 补 `main`、`meta description`、相关推荐、前后跳转；P1 把演示模块抽成可复用组件规范，沉淀为后续交互题模板；P2 为移动端增加更明确的点击反馈与重置操作。验收：用户先理解题，再做操作，且重复进入或多实例渲染时不会因 SVG `id` 冲突导致显示异常。
- `problems/heaven-hat-game.html` : P0 降低首屏噪声，只保留题干摘要、主图和主行动按钮，把复杂动态覆盖关系放到步骤区逐步揭示；P0 拆分超长内联 CSS 和 SVG 逻辑到独立资源；P1 优化移动端图文联动，确保图示不会挤压正文；P1 副标题改为“帽子推理 / 公共知识 / 递归推断”等用户可理解关键词；P2 增加相关推荐与学习路径。验收：首屏不再像说明书，动态内容不会压缩题意理解空间。

## 4. 题型主题设计库

### 数学推导型

- 适用题目 : `problems/age-puzzle-1.html`、`problems/age-puzzle-2.html`、`problems/bacteria-split.html`、`problems/guess-number-144.html`、`problems/clock-hands-coincide.html`
- 主题关键词 : 递推、倍增、时间、数量变化、公式推导
- 配色方向 : 深蓝、琥珀、青金、荧光线条；避免过多花哨纹理，突出“推演感”
- Hero 建议 : 题目摘要卡 + 时间轴或数字增长图形 + 一个关键公式提示
- 模块建议 : 公式卡、变量关系图、步骤演进卡、关键转折高亮
- 动效建议 : 数值递增、路径推进、公式淡入，不做大幅飘动动画

### 博弈决策型

- 适用题目 : `problems/pirate-gold.html`、`problems/gold-bar.html`、`problems/king-prophet.html`、`problems/five-prisoners.html`
- 主题关键词 : 角色、投票、收益、顺位、策略选择
- 配色方向 : 深酒红、金色、铜色、深海蓝；营造“决策压力”和“利益分配”氛围
- Hero 建议 : 角色卡阵列 + 顺位关系 + 核心冲突摘要
- 模块建议 : 回合卡、投票流程、收益表、策略分叉卡
- 动效建议 : 角色高亮切换、投票流向、顺位递进

### 空间想象型

- 适用题目 : `problems/ballerina-mirror-room.html`、`problems/ring-rolling-rotation.html`、`problems/portia-portrait-1.html`、`problems/portia-portrait-2.html`
- 主题关键词 : 镜像、旋转、视角、空间关系、几何直觉
- 配色方向 : 冷银、蓝紫、玻璃光、镜面高光
- Hero 建议 : 主示意图先行，文字做辅助；优先让用户看见空间关系
- 模块建议 : 视角切换图、二维到三维过渡图、空间分层卡
- 动效建议 : 轻微旋转、镜像翻转、路径描边；控制速度，避免眩晕

### 真假话与身份推理型

- 适用题目 : `problems/strange-villages.html`、`problems/birthday-deduction.html`、`problems/eleven-disciples.html`
- 主题关键词 : 身份、真假、线索排除、条件筛选
- 配色方向 : 墨绿、暗金、象牙白、对比态红蓝
- Hero 建议 : 人物身份卡 + 已知条件摘要 + 一条关键线索提示
- 模块建议 : 线索面板、身份卡、排除高亮、真假状态标签
- 动效建议 : 状态切换、条件点亮、排除线收束

### 互动模拟型

- 适用题目 : `problems/soda-bottle.html`、`problems/blind-socks.html`、`problems/three-bulbs.html`
- 主题关键词 : 操作、反馈、状态变化、即时结果
- 配色方向 : 高对比主题色 + 发光强调色；保证按钮和状态足够清晰
- Hero 建议 : 先用一张主图或状态组件解释规则，再放主操作入口
- 模块建议 : 控制面板、状态计数、操作日志、结果回放、洞见总结
- 动效建议 : 必须直接反馈操作结果，例如数量变化、亮灭切换、交换过程

### 路径与过程型

- 适用题目 : `problems/bird-train-distance.html`、`problems/last-athlete-standing.html`、`problems/monkey-bananas.html`
- 主题关键词 : 路径、速度、追赶、来回、阶段变化
- 配色方向 : 夜空蓝、速度橙、轨迹青
- Hero 建议 : 路径示意主图 + 起点终点 + 关键约束提示
- 模块建议 : 轨迹线、过程分段卡、速度对比条、路径节点
- 动效建议 : 路径推进、节点点亮、位置移动

## 5. 单页制作 SOP

- 第一步，提炼题感 : 写清楚这题的核心机制、核心意象、用户最难理解的点，各用一句话描述。
- 第二步，确定主题 : 为页面确定主色、辅助色、质感方向、图形母题，例如气泡、砝码、帽子、轨迹、宝箱。
- 第三步，搭 Hero : 首屏必须同时回答“这题讲什么”“难点在哪”“为什么值得继续看”。
- 第四步，选模块 : 从题型主题库里选择 3 到 5 个最适合的模块，而不是把所有模块都堆上去。
- 第五步，补统一骨架 : 接入 `main`、面包屑、结论区、延伸区、推荐区、上一篇下一篇。
- 第六步，做题意联动 : 至少增加 1 个直接帮助理解题目的主题模块，例如模拟器、示意图、关系图、状态切换。
- 第七步，做移动端检查 : 确认首屏不拥挤，关键图形不会溢出，按钮能点，表格可读。
- 第八步，做文案收束 : 副标题、标签、结论文案、洞见文案都要从“题感”出发，不写空泛描述。

## 6. 页面组成标准

- Hero 区 : 必须包含题目标题、题意摘要、主题图形或主视觉、进入阅读或互动的主引导。
- 题干区 : 必须把题目条件重新整理成更易读的版本，必要时拆成列表、关系图或条件卡。
- 推理区 : 必须是页面主体，可使用步骤卡、表格、流程图、时间轴、公式卡等形式。
- 结论区 : 必须有明确答案、结论解释、是否存在边界条件或例外情况。
- 洞见区 : 必须总结“这题到底考什么”“最容易错在哪里”“通法是什么”。
- 串联区 : 必须提供相关推荐、上一篇下一篇、返回题库或专题入口。

## 7. 主题化实施清单

- 视觉层 : 是否有专属主题色、专属图形母题、专属背景或纹理点缀。
- 内容层 : 是否有与题目机制匹配的解释模块，而不是只换标题文案。
- 交互层 : 是否至少有 1 个与题意相关的互动或状态反馈设计。
- 结构层 : 是否仍然遵守全站统一骨架，避免因为主题化而失控。
- 代码层 : 是否把通用规则留在 `assets/`，把个性化变量留在页面内，避免再次散乱。

## 8. 验收清单

- 首屏验收 : 用户进入页面 3 到 5 秒内，能理解题目大意并感受到题目主题。
- 主题验收 : 去掉正文后，仅看 Hero、配色、图形和模块标题，仍能判断题目大致类型。
- 阅读验收 : 页面信息密度分布合理，不会出现首屏过满、结尾过空或正文大段无停顿。
- 交互验收 : 所有点击控件、折叠区、步骤推进和模拟操作都可键盘触达。
- 移动端验收 : 主图不溢出、主按钮可点、表格可滚动或转换视图、首屏高度可控。
- 复用验收 : 该页沉淀出的主题变量、组件或模块，后续至少还能复用到 1 个同类题目。

## 9. blind-socks.html 专项页面方案

### 页面定位

- 页面角色 : 作为“互动模拟型题页”的标杆页，承担后续同类题目的主题模板示范作用。
- 题目气质 : 不是“袜子”本身，而是“黑暗中依靠触觉与对称性消除不确定性”。
- 核心主题词 : 黑暗、触觉、配对、撕开商标、平分、均衡、对称。
- 目标体验 : 用户先感受到“看不见”的困境，再通过操作理解“平分每一对”为什么必然正确。

### 当前页面优点

- 已有较强的主题雏形 : 暗色背景、布料质感、盲人视角切换、袜子 SVG、分袜演示都已经具备基础。
- 已有可复用交互 : `逐对平分`、`重置`、`盲人视角`、统计面板，具备做成互动模板的潜力。
- 已有题意契合方向 : “触觉 + 商标纸 + 对称性”这条主线是对的，只是还没有彻底拉开层次。

### 当前页面主要短板

- 首屏还不够像“谜题入口” : 现在题干有了，但缺少“为什么商标纸是关键线索”的视觉强调。
- 信息架构偏平 : 题干、解析、演示、洞见都堆在同一段落节奏里，阅读顺序不够强。
- 交互说明偏弱 : 用户进入演示前，不一定马上知道“为什么要先切换盲人视角，再逐对平分”。
- 结果表达还可以更戏剧化 : 目前统计正确，但“均衡结果的必然性”没有被强烈地可视化呈现。

### 主题视觉方向

- 主视觉母题 : 袜子配对、商标纸连接、盲文点阵、布料纹理、对称分割线。
- 配色方向 : 保留当前暗金 + 炭黑 + 象牙白体系，把 `商标纸金色` 作为唯一高亮主线。
- 光效策略 : 只在“关键线索”和“最终均衡结果”上发光，不做大面积炫光。
- 字体气质 : 保持现在的古典衬线基调，但关键信息要更偏理性与秩序感。
- 情绪目标 : 从“不确定”逐步过渡到“秩序显现”和“结果必然”。

### 建议信息架构

- 第一屏 `Hero` : 题目标题 + 一句副标题 + 题意摘要卡 + “关键线索提示卡”
- 第二屏 `困境说明区` : 用 3 张小卡说明“看不见颜色”“材质完全一致”“唯一可感知的是商标纸”
- 第三屏 `解析区` : 保留三步逻辑，但改成更强的步骤流和状态指引
- 第四屏 `演示区` : 明确分为“先切盲人视角 -> 再逐对平分 -> 看统计结果”三步
- 第五屏 `结果区` : 用强对称可视化展示 A/B 最终都得到 4 黑 4 白
- 第六屏 `洞见区` : 总结“从分类问题转化为平分问题”
- 第七屏 `延伸区` : 增加相关推荐、同类题、上一篇下一篇

### Hero 详细方案

- 标题建议 : 保留 `盲人分袜`
- 副标题建议 : 从 `The Symmetry of Uncertainty` 改为更贴题的双语表达，例如 `黑暗中的对称解法 | The Symmetry of Uncertainty`
- 题意摘要卡 : 缩短成 3 个条件块，不用整段描述
- 关键线索卡 : 单独强调 `每对袜子由商标纸连接`，并给一个小图标或小示意图
- Hero 主视觉 : 将当前大水印升级为“左右两只袜子由商标纸连接”的主构图，中央加一条对称分割线
- Hero 引导按钮 : 增加一个 `先看关键线索` 或 `开始分袜演示` 的跳转入口

### 困境说明区方案

- 模块目标 : 让用户在进入推理前，真正理解这题难点不是颜色，而是“不可辨识条件下如何公平分配”
- 卡片 1 : `颜色不可见`，文案强调盲人无法区分黑白
- 卡片 2 : `触感相同`，文案强调材质、尺寸完全一致，不能靠手感分类
- 卡片 3 : `唯一线索`，文案强调商标纸把同色两只袜子连成一个最小单位
- 视觉建议 : 三张卡从弱到强递进，最后一张使用高亮边框和微发光

### 解析区方案

- 保留现有三步逻辑，但每一步要明确“解决了什么问题”
- 第一步标题建议 : `先找配对单位，不找颜色`
- 第二步标题建议 : `逐对撕开，机械平分`
- 第三步标题建议 : `局部平分，整体必然均衡`
- 交互建议 : 步骤卡除了高亮，还要显示“前置问题 -> 当前动作 -> 得到结果”
- 视觉建议 : 三步之间增加细连接线，让用户感觉是在走一条确定性的推理路径

### 演示区方案

- 增加演示前说明条 : 明确告诉用户“推荐先开启盲人视角，再点击逐对平分”
- 左侧主区 : 袜子配对可视化区域，强调商标纸连接和撕开动作
- 右侧统计区 : 显示 A/B 当前拿到的袜子数量与颜色分布
- 增加状态阶段提示 : `未开始`、`正在平分第 N 对`、`分配完成`
- 增加结果回放入口 : 分配完成后可高亮回看每一对是如何平分的
- 增加一次性演示模式 : 除 `逐对平分` 外，再加 `自动完成`，用于快速验证结论

### 关键交互建议

- `盲人视角` 开关 : 从纯视觉切换升级为“教学开关”，开启后附带一句提示“你现在不能依赖颜色，只能依赖配对关系”
- `逐对平分` 动效 : 要突出“先撕开商标纸，再左右分发”的动作顺序
- `统计卡` 反馈 : 完成每一步后，对应 A/B 面板短暂闪亮，强调“双方同步增加”
- `最终结果` 动效 : 第 8 对完成后，A/B 同时出现金色对称高亮，强化“公平是必然结果”

### 结果区方案

- 单独从演示区下方拆出 `最终结果` 区，不要只靠统计面板承担总结职责
- 结果主文案 : `每人最终都得到 4 只黑袜 + 4 只白袜`
- 结果副文案 : `他们不需要看见颜色，只需要平分每一个同色单位`
- 可视化建议 : 左右两列对称展示 A/B 结果，中间是“=”
- 强调方式 : 用对称布局、统一动画和统一数量标签来表现“结果不是巧合，而是结构必然”

### 洞见区方案

- 当前洞见方向是对的，但可以再拆成两层
- 第一层 : `这题考什么`，回答“考的是把分类问题转为平分问题”
- 第二层 : `这种思路还能用在哪`，引出对称性、冗余信息、最小单位分割
- 可增加一个 `通法卡片` : 当不可识别属性在最小单位内部完全相同，可先按单位平分，再在总量上得到均衡

### 推荐新增模块

- `关键线索卡` : 强调商标纸是破局点
- `困境三联卡` : 把条件重写得更好懂
- `结果对称墙` : 最终 A/B 分配结果的大视觉总结
- `通法卡片` : 抽象出方法论，提升页面深度
- `相关推荐区` : 推荐 `soda-bottle.html`、`three-bulbs.html`、`hat-color-*` 一类同样依赖规则与信息限制的题

### 代码与组件拆分建议

- 页面专属 Token : 保留 `--sock-black`、`--sock-white`、`--tag-color`，再补 `--theme-primary`、`--theme-glow`、`--theme-divider`
- 可复用组件 1 : `规则演示面板`，后续可给 `three-bulbs.html`、`soda-bottle.html` 复用
- 可复用组件 2 : `结果对称统计卡`，后续可给公平分配类或双人博弈类题目复用
- 可复用组件 3 : `关键线索提示条`，后续用于所有“某个小条件决定破局”的题
- 风险点 : 当前 SVG `pattern id` 仍有重复风险，后续正式改代码时要改成实例唯一命名

### 实施顺序建议

- 第一步 : 重做信息架构，拆出 Hero、困境区、解析区、演示区、结果区、洞见区
- 第二步 : 强化 Hero 主视觉与关键线索卡
- 第三步 : 重构演示区说明与状态流转
- 第四步 : 单独补最终结果区和通法卡片
- 第五步 : 再处理相关推荐、上一篇下一篇和移动端细节

### 本页验收标准

- 用户在不操作页面时，也能在首屏理解“商标纸是关键线索”
- 用户在开始操作前，能知道推荐流程是“开盲人视角 -> 逐对平分 -> 看结果”
- 用户在演示结束后，能明显感知“公平结果来自结构对称，而不是运气”
- 用户在离开页面前，能记住一句方法论，而不只是记住答案
